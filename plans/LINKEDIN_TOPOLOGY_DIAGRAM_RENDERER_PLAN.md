# LinkedIn Topology Diagram Renderer Upgrade

## Summary
Upgrade the LinkedIn content team’s architecture visuals from stage-card diagrams to traditional topology diagrams with a left-side architecture canvas and right-side numbered legend, matching the provided reference style. Architecture prompts should default to this new renderer; the current column/card renderer remains available for workflow/process explainers.

## Key Changes
- Save this plan as `plans/LINKEDIN_TOPOLOGY_DIAGRAM_RENDERER_PLAN.md`.
- Add a new deterministic renderer, `architecture-topology-v1`, for architecture diagrams generated by `/linkedin-content`.
- Add a richer topology spec alongside the existing column spec:
  - architecture groups/boundaries such as UI, API/control plane, agent teams, memory/data, external gates, analytics, and approval layer
  - positioned nodes with icons, short labels, and optional group membership
  - edges with labels and styles such as solid, dashed, and bidirectional
  - right-side numbered legend cards with short explanations
  - provenance and quality metadata
- Make architecture prompts default to the topology renderer:
  - `system_architecture` uses topology + legend
  - `agent_workflow`, `data_flow`, and `approval_gates` may keep the current workflow/card renderer unless the prompt explicitly asks for system architecture
- Render with Playwright to a fixed LinkedIn-safe PNG:
  - 1600x900 canvas
  - approximately 65% topology map and 35% legend
  - subtle dotted background, nested boundaries, compact service icons, connector lines, and restrained typography
  - no body text overflow, no all-bold labels, no cramped legend cards
- Add an internal icon vocabulary for Job Search OS concepts:
  - Next.js/App UI, API routes, Jolene, Operating Loop, Email Ops, LinkedIn Content Team, Memory Pack, Prisma/Postgres, AgentRun, Analytics, Playwright Screenshots, Privacy Review, Approval Gates, LinkedIn Publishing
- Extend diagram QA:
  - validate node bounds, edge label length, legend card overflow, contrast, topology density, missing provenance, and excessive text
  - block publishable diagram status if the topology or legend overflows
- Update `/linkedin-content` visual review UI metadata:
  - show renderer as `architecture-topology-v1`
  - show diagram kind, topology quality score, warnings, provenance, and selected-state rationale
  - continue selecting the highest-scoring deterministic technical diagram by default

## Public Interfaces
- No request-body change for `POST /api/linkedin-content/drafts`.
- Extend visual asset metadata with topology-specific fields:
  - `renderEngine: "architecture-topology-v1" | "staff-engineer-html-v1"`
  - `layoutKind: "topology_legend" | "workflow_columns"`
  - `topologySpec`
  - `qualityReview.topology`
  - `qualityReview.legend`
- Keep generated assets under `public/generated/linkedin-content`.
- Keep AI image polish optional and non-authoritative; exact technical architecture diagrams remain deterministic.

## Test Plan
- Unit/source tests:
  - architecture prompts produce a topology spec, not only column specs
  - topology spec includes groups, nodes, edges, legend cards, and provenance
  - architecture drafts select the topology diagram as the primary deterministic asset
  - workflow prompts still support the existing stage/card renderer
  - QA rejects overflowing node labels, overflowing legend cards, missing provenance, weak contrast, and overly dense topology
- Route tests:
  - generating a draft with “system architecture diagram” returns a visual asset with `renderEngine: "architecture-topology-v1"`
  - failed topology rendering preserves the draft but marks the visual asset as `NEEDS_REVIEW`
- UI/source tests:
  - `/linkedin-content` displays topology renderer metadata and quality warnings
  - visual cards still separate technical diagrams, AI polish variants, and app screenshots
- Verification:

## Assumptions
- Chosen style: topology canvas plus numbered legend, matching the provided reference.
- V1 uses generic Job Search OS/product-system icons, not AWS-specific icons, unless the diagram is explicitly about AWS/cloud infrastructure.
- The new renderer is for LinkedIn content visuals only; it does not create a general documentation library yet.
- Current privacy, provenance, edit, approval, and publishing gates stay unchanged.
